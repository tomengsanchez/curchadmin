Backend stack
Custom PHP MVC framework, not Laravel/CodeIgniter.
Core framework lives under Core/ (router, base controller, DB, auth, CSRF, logger, mail, migrations, Helpers, SystemDebug, LoggingPDO/LoggingStatement for debug).
Requests enter through public/index.php (front controller) and are routed via Core\Router to App\Controllers\... methods.
Config: copy config/database-sample.php to config/database.php and config/app-sample.php to config/app.php (optional base_url).
API (REST)
REST-style API under /api/: auth (login, me, logout), dropdowns (projects, profiles, structures, notifications), dashboard and grievance dashboard aggregates, settings, profile/structure CRUD, history. API controllers live under App\Controllers\Api\. Authentication via Bearer token (App\ApiToken) when no session; tokens stored hashed in api_tokens table (migration 020). Use requireAuthApi() in API actions for 401 JSON on unauthenticated requests.
Auth and security
Session auth (Core\Auth); optional idle session timeout (user_logout_after_minutes in Security Settings; 0 = disabled). Two-factor authentication (2FA): optional email-based 2FA (enable_email_2fa, 2fa_expiration_minutes in app_settings); routes /login/2fa and /login/2fa/verify. User sessions: user_sessions table (migration 023) tracks device/session per user; App\UserSession on login and each request; /account/sessions lists active sessions and allows "logout other devices" (SessionsController).
Frontend stack
Server-rendered PHP views, no SPA framework.
Uses Bootstrap 5, jQuery, and Select2 (with a Bootstrap 5 theme) for UI and enhanced form controls.
Main layout supports:
- Sidebar or top-navigation layouts (per-user via UserUiSettings).
- Optional “super mobile-friendly” mode, which on small screens:
  - Stacks Bootstrap grid columns vertically.
  - Makes tables more compact with horizontal scrolling.
  - Turns the sidebar into an overlay drawer with a hamburger toggle and backdrop.
Application structure
App\Controllers\ – HTTP and API controllers (e.g. dashboard, profile, structure, grievance, users, settings, security, notifications, etc.). – PDO-based models, no heavy ORM; mostly one model per table/domain.
App\Views\ – module-based PHP templates with a shared layout (layout/main.php) and partials (pagination, toolbars, sidebars).
Clear MVC pattern with role/capability-based access checks in controllers and menus.
Export: CSV export from Profile, Structure, Grievance, Users, and Library. Select Columns and Export dialogs show all available fields (ListConfig::getExportColumns). CsvExporter streams CSV; Grievance::listForExport resolves lookup names.
Database layer & schema
DB config via config/database.php, used by Core\Database (PDO singleton).
Schema defined by PHP migration scripts under database/migration_*.php and seeders under database/seed_*.php. Migrations: run via php cli/migrate.php; status via --status; rollback via --rollback or --rollback --steps=N (LIFO, requires callable 'down' in each migration). Cron: run php cli/send_queued_emails.php periodically (e.g. every 1–5 min) to send notification emails from email_queue.
Main domains: users/roles/capabilities, projects, profiles, structures, grievances + grievance status/history, app/user settings, notifications, audit log, API tokens, user_sessions (active session tracking).
Security-related schema additions:
- app_settings: stores security and UI configuration (2FA, login throttling, password policy, email settings, UI preferences, etc.).
- users.password_changed_at: used for password expiry checks.
- user_password_history: per-user history of password hashes for enforcing “last N passwords” reuse rules.
- user_sessions (migration 023): user_id, session_id, user_agent, ip_address, created_at, last_activity_at, revoked_at; for "where you're logged in" and remote logout.
- api_tokens (migration 020): token_hash, user_id, expires_at for REST Bearer auth.
Development and assets
DevClock: optional simulated current time for development (System > Development). Serve routes: /serve/structure, /serve/profile, /serve/grievance, /serve/grievance-card-attachment, /serve/app-logo for images, attachments, and app logo.
Grievance progress levels and escalation (latest):
- grievance_progress_levels is project-scoped (migration_026): project-specific levels are supported via project_id; default fallback levels use project_id = NULL (Level 1/2/3).
- Escalation timing does not reset on note-only updates. Timer resets only on real status/level transition; logic uses start of current in-progress segment (not MAX(created_at)).
- Dashboard/API are project-aware for level labels and escalation rows; mixed-project views prefix labels with project name when needed.
- Remap tools:
  - UI: Grievance > In Progress Stages > "Re-map existing records"
  - CLI: php cli/remap_progress_levels.php [--project=<id>]
- Seeder options:
  - php database/seed_profiles_structures_with_project_levels.php (project-specific stage names per project)
  - php database/seed_profiles_structures_with_default_levels.php (default-only levels)
  - php database/seed_progress_levels_default.php (only Level 1/2/3 defaults)
